- [x] `@pharmax/crypto` — envelope encryption with per-tenant KEK, **landed before any PHI table**. Implements the full primitive set the Phase 2 patient/prescription/order tables will depend on. **Canonical AAD encoding** (`crypto.v1` versioned, NUL-separated, sorted-key) over the record binding `{tenantId, table, column, recordId}` — every binding field demonstrably influences the AAD so a ciphertext moved between rows (different recordId, different column, different table, or different tenant) fails authentication on decrypt; empty fields and embedded NUL bytes are rejected at encode time. **Ciphertext envelope** is a JSON object (`{v: 1, alg: "AES-256-GCM", kek, wDek, iv, ct, tag}`) stored as a Prisma `Json` column; `parseEnvelope` is the choke point that rejects malformed shapes before the cipher sees them, and `ct === ""` is legitimate (zero-byte plaintext for empty PHI fields like `address_line_2`). **`KmsAdapter`** interface separates the seam between this package and the key store; production binds AWS KMS, dev/test binds `LocalKmsAdapter` which derives per-tenant KEKs via HKDF-SHA-256 (`info = "pharmax.kek.v1.<tenantId>.v<n>"`), rotates KEKs in-place via `rotateKek({tenantId})` (kid version bumped; historical wrapped DEKs continue to unwrap — the rotation guarantee), and derives per-tenant per-purpose 32-byte search keys (`info = "pharmax.search.v1.<tenantId>.<purpose>"`) that are decoupled from KEK lifecycle so blind-index columns survive KEK rotation. KEK-wrap is AES-256-GCM with random IV per wrap; the wrapped DEK is exactly 60 bytes (12 IV + 16 tag + 32 ciphertext). **`encryptField`/`decryptField`** generate a fresh DEK per call, mix the AAD into GCM, zero the plaintext DEK after use, and surface a GCM tag-verification failure as `AuthorizationError(AAD_MISMATCH)` (not a generic decrypt error — the security signal is "someone moved a ciphertext between rows"). Re-encrypting the same plaintext produces a different envelope (`iv`, `wDek`, `ct`, `tag` all change) so storage doesn't leak "these two rows hold the same value". **`blindIndex`** is HMAC-SHA-256 over `normalize(value)` keyed by the per-tenant per-purpose search key, base64url-encoded (43 chars) — deterministic for queryability, separated across tenants AND columns AND tables, returns `null` for empty-normalized values (caller leaves the column NULL rather than matching every NULL-equivalent row). Built-in normalizers: `normalizeForBlindIndex` (lowercase + trim + NFD-strip-combining-marks + whitespace collapse, so "Café" matches "Cafe" matches " JANE " matches "jane") and `normalizePhoneForBlindIndex` (digits-only, last-10). **`planCryptoShred`** is a pure, no-I/O intent boundary returning `{nextValue: null, reason, ...binding}` — callers apply the plan inside a command-handler transaction so the storage write, audit row, and outbox event commit atomically; reason vocabulary is frozen (`RIGHT_TO_BE_FORGOTTEN`, `TENANT_OFFBOARD`, `DATA_RETENTION_EXPIRY`, `PATIENT_DECEASED_RECORD_CLOSE`). **`configureCrypto`** is the boot singleton; reading without it throws `InternalError(CRYPTO_NOT_CONFIGURED)` (silence here would let a PHI write proceed unencrypted, the worst possible failure mode). **94 contract tests across 7 files** cover AAD determinism + per-field uniqueness + validation (13), envelope parse/serialize/guard (12), LocalKmsAdapter wrap/unwrap/rotation/tenant-isolation/search-key/empty-seed-rejection (22), end-to-end encrypt/decrypt with UTF-8/CJK/emoji/empty-plaintext roundtrip + AAD-mismatch on every binding field + cross-tenant ciphertext rejection + bit-flip tamper detection on `ct` and `tag` + IV-length guard + rotation continuity + JSON wire round-trip (15), blind-index determinism + normalization (case/accent/whitespace) + tenant/column/table separation + phone normalizer + null on empty + base64url shape (18), shred plan + reason validation + frozen-registry guard (10), configure singleton (4). **Wired at boot** in `apps/web` (`instrumentation.ts` → `src/server/bootstrap.ts`), `apps/worker` (`main.ts`), and `scripts/bootstrap-org.ts` — every entry point hard-fails in `NODE_ENV=production` because `LocalKmsAdapter` MUST be replaced by `AwsKmsAdapter` before prod promotion. Env contract: `PHARMAX_LOCAL_KMS_SEED` (≥32 chars, validated by Zod); apps/web and apps/worker MUST agree on this value or rows encrypted by one process are undecryptable by the other.

## Phase 2 — Order workflow

- [x] **Postgres Row-Level Security baseline migration** (`20260522060000_rls_baseline`) — first Phase 2 migration, lands before any PHI table. Creates two Postgres roles (`pharmax_app` subject to RLS for normal runtime, `pharmax_system` with `BYPASSRLS` for the bootstrap path) — both `NOLOGIN` at creation; ops attaches credentials out of band. Enables AND **forces** RLS on all 18 tenant-scoped tables (FORCE means even the table owner is policy-checked, so a curious DBA in psql sees the same isolation runtime sees). Installs one `tenant_isolation` policy per table whose predicate is `current_setting('pharmax.system_context', true) = 'on' OR <tenant>` where `<tenant>` is `id = NULLIF(current_setting('pharmax.organization_id', true), '')::uuid` for `organization` and `"organizationId" = ...` for the rest; `NULLIF` collapses the empty-string GUC clear into a SQL NULL so an unset GUC fails the predicate (fail-closed). `audit_log` gets policy-level immutability: only `tenant_isolation_select` (FOR SELECT) and `tenant_isolation_insert` (FOR INSERT) policies — UPDATE/DELETE have no policy and are therefore denied by RLS, AND a redundant `REVOKE UPDATE, DELETE` from both app roles makes that immutability survive any future policy-table refactor. Session-GUC writers (`applyTenancySessionGuc` / `applySystemSessionGuc` / `clearSessionGuc`) live in `@pharmax/tenancy` and are invoked by the command bus as the FIRST statement inside every transaction (every value, including the empty-string clear and the literal `'on'`, is passed as a bound parameter — no string interpolation, no injection surface). Migration linter `scripts/check-migration-rls.ts` walks every migration in apply order and rejects any `CREATE TABLE` not paired with `ENABLE ROW LEVEL SECURITY` + `CREATE POLICY` (either in the same migration or any later one) and not listed in `prisma/migrations/rls-exempt.txt`; any `DISABLE ROW LEVEL SECURITY` is a hard failure regardless of exemption. Four documented exemptions today (`permission`, `role_permission`, `clinic_site`, `stripe_webhook_event`) each with the architectural justification in a code-reviewed comment. See `ARCHITECTURE_PRINCIPLES.md` §B.1 + §C.1.
- [x] **Audit hash-chain writer + verifier** — `audit_log` gained three columns in migration `20260522190000_audit_chain` (`prev_hash` BYTEA nullable, `entry_hash` BYTEA NOT NULL, `seq` BIGINT NOT NULL) backed by a `(organizationId, seq)` unique index — the per-tenant integrity guarantee. The migration backfills `seq` (over `occurredAt`, `id`) and a sentinel `entry_hash = SHA-256("legacy:<id>")` for any pre-chain rows so the NOT NULL constraint is safe to enforce. `audit_chain_state` (one row per organization, RLS + FORCE-RLS protected, `SELECT/INSERT/UPDATE` only on the app/system roles) holds the latest `(latestHash, latestSeq, updatedAt)` so the writer never scans `audit_log` to find the head. SQL helper `audit_chain_lock_key(uuid)` derives a stable BIGINT advisory-lock key from the org UUID with a `0x6175646974636861` ('auditcha') salt to avoid collision with Prisma's migration lock; the function is `IMMUTABLE STRICT SECURITY INVOKER` and granted `EXECUTE` only to the two app roles. **`@pharmax/audit`** ships `writeAuditLogInTx(tx, input)` — the only supported audit insert path — which acquires `pg_advisory_xact_lock(audit_chain_lock_key(orgId))` as the FIRST statement, reads the chain head, computes `entryHash = SHA-256(canonical(prevHash, organizationId, seq, action, resourceType, resourceId, actorUserId, scope, metadata, occurredAt))`, inserts the audit row, and upserts the chain head — all inside the caller's tx so a rollback leaves the head untouched. Genesis insert path: head NULL → seq=1, prevHash=null; chained insert: seq=head.latestSeq+1, prevHash=head.latestHash. `@pharmax/command-bus` `createAuditLogInTx` delegates to the chain writer (handlers' audit shape is unchanged; the chain is invisible to command authors). **Canonical TLV encoder** is 1-byte field tag + 4-byte big-endian length + value bytes, with a NULL sentinel (`length=0xFFFFFFFF`) that distinguishes "field present but null" from "field present and empty"; field order is fixed and documented as version-controlled; JSON columns are canonicalized via sorted-key stringify matching `@pharmax/command-bus`'s hash encoder; bigint payloads inside scope/metadata are rejected at encode time (callers must string-encode); seq is fixed 8-byte big-endian uint64. `computeAuditEntryHash` is pure and exposed so the verifier and signing job re-derive hashes from persisted bytes — verifier byte-equivalence is the chain integrity guarantee. **`verifyChain(source, args)`** walks a tenant's chain in ascending seq order against an injected `ChainSource` (in-memory for tests, Prisma-backed in production) and fails fast on the first inconsistency with `InternalError(AUDIT_CHAIN_BROKEN)` carrying the offending seq and a reason tag (`entryHash mismatch`, `prevHash` mismatch, or `seq gap`); supports `startSeq`/`endSeq` bounds and a `startPrevHash` checkpoint for resumed walks. **47 contract tests across 3 files** — encoder determinism + per-field uniqueness (changing any of prevHash/orgId/seq/action/resourceType/resourceId/actorUserId/scope/metadata/occurredAt by even 1ms flips the hash) + TLV byte layout + seq encoding + JSON canonicalization + seq bounds (19), writer call ordering (lock → findUnique → audit.create → chainState.upsert) + advisory-lock parameter binding (orgId bound, not interpolated) + genesis vs chained paths + multi-insert chain continuity (13), verifier happy paths + tamper detection (mutating entryHash, action, scope, occurredAt by 1ms, or prevHash all trip AUDIT_CHAIN_BROKEN at the right seq) + seq gap detection + per-tenant isolation (verifying tenant A succeeds while tenant B's chain in the same source is broken) + resume-from-checkpoint with right/wrong startPrevHash (15). **Command-bus contract tests** additionally pin (a) the advisory lock fires AFTER the tenancy session GUC and BEFORE the audit_log create, (b) the lock-key derivation passes orgId as a bound parameter, (c) the genesis-insert audit row carries seq=1 + prevHash=null + a 32-byte entryHash, and (d) `audit_chain_state` is upserted in the same tx. **Daily Merkle-root signing job deferred** — when it lands it will read `audit_chain_state` rolling forward, build a per-day Merkle tree per tenant, sign with a KMS asymmetric key, and write the manifest to an S3 Object Lock bucket; see `ARCHITECTURE_PRINCIPLES.md` §B.3 + §C.4.
- [x] Patient, Provider, Prescription, Order, OrderLine schema with PHI envelope columns (`*Enc Json`) + blind-index columns (`*Bi String`, base64url HMAC-SHA-256, 43 chars) — single migration `20260523190000_phase2_patient_rx_order` that introduces seven new enums (`PatientStatus`, `ProviderStatus`, `PrescriptionStatus`, `OrderStatus` with the 11 workflow + 5 exception states, `OrderPriority`, `OrderLineStatus`, `IntakeSourceKind`), the five new tables, all indexes, FK relations, and in the same file promotes the three baseline placeholder columns to real FKs (`command_log.targetOrderId`, `order_event.orderId`, `invoice_line.orderId` — all `ON DELETE RESTRICT` so the audit trail can never lose its anchor). Every new tenant-scoped table comes under the RLS regime from `20260522060000_rls_baseline` (GRANT to `pharmax_app`/`pharmax_system`, `ENABLE` + `FORCE` row level security, single PERMISSIVE `tenant_isolation` policy on the `pharmax.system_context` / `pharmax.organization_id` GUC pair, applied via a DO block keyed on a `new_tables text[]` array) and is verified by `scripts/check-migration-rls.ts`. Patient identity is scoped to `(organizationId, clinicId)` by design — the same human at two clinics is two patient rows, because cross-clinic linkage is a deliberate audited admin action, not an implicit join. PHI columns required for identity (`firstNameEnc`, `lastNameEnc`, `dateOfBirthEnc`) are NOT NULL; demographic and contact PHI is nullable so intake can proceed against partial data. The DOB has two blind-index columns (`dobBi` for full-date exact match, `dobYearMonthBi` for fuzzy month-of-birth typo recovery) under distinct purposes so search keys can't cross-contaminate. Crypto-shred tombstone (`patient.cryptoShreddedAt`) is the column the right-to-be-forgotten flow writes when the DEKs are destroyed at the KMS; the row stays for FK integrity, the ciphertexts become unreadable. - [ ] **SLA semantics for the 6 commands in `KNOWN_NON_SLA_COMMANDS`** (`packages/sla/src/interval-recorder.ts`) — `RejectPV1`, `RejectFinalVerification`, `PlaceHold`, `ReleaseHold`, `CancelOrder`, `ReopenForCorrection` are explicit no-ops today (the order's currently-open `OrderStageInterval` is unaffected when they fire). Each one mutates order workflow state in ways that almost certainly **should** affect SLA timing, but each is a **policy call** about what "stage time" means to a pharmacy manager looking at a dashboard, and the policy is undecided. Resolving each: (a) write the decision below, (b) move the command name out of `KNOWN_NON_SLA_COMMANDS` into either `COMMAND_STAGE_INTERVAL_TRANSITION` (close + open) or `COMMAND_STAGE_INTERVAL_CLOSE_ONLY` (close-only), (c) add any new `OrderStageIntervalKind` enum values + migration, (d) update the command handler to pass the right `commandLogId` / `actorUserId` to the recorder, (e) add tests in the relevant command package mirroring `start-typing.test.ts`'s recorder assertions. **Open questions per command:** **(1) `RejectPV1` / `RejectFinalVerification`** — when a pharmacist rejects, the order routes to an exception queue and a tech reworks the rx; should the `PV1_ACTIVE` / `FINAL_VERIFICATION_ACTIVE` interval **close** (and a new `WAIT_AFTER_PV1_REJECT` / `WAIT_AFTER_FINAL_REJECT` enum value open until the next `StartPV1` / `StartFinalVerification`)? Or does the rejected stage's active interval **stay open** through the rework loop so that managers see the full elapsed pharmacist time as one continuous block? Trade-off: closing gives clean per-decision-attempt time but inflates "first-pass" counts; leaving open buries rework cost inside the "PV1 time" metric. **Recommended:** close + new `WAIT_AFTER_*_REJECT` kinds — dashboards can re-aggregate but they cannot un-bury a hidden rework loop. **(2) `PlaceHold` / `ReleaseHold`** — a held order is paused (waiting on prescriber callback, missing info, etc.). Should the currently-open interval **close** on `PlaceHold` and a new `HOLD_ACTIVE` interval open until `ReleaseHold`? If we do nothing, every held order accumulates "active stage time" while sitting idle and **looks like an SLA breach** that the pharmacy cannot fix. **Recommended:** add `HOLD_ACTIVE` enum value; `PlaceHold` is a transition (close current, open `HOLD_ACTIVE`); `ReleaseHold` is a transition (close `HOLD_ACTIVE`, open whatever stage the release returns to — which the command already knows because it reads `cancelledFromStatus` / equivalent off `order_hold`). **(3) `CancelOrder`** — terminal; should close any currently-open interval and not open a successor (same shape as `ConfirmShipment`, which is already in `COMMAND_STAGE_INTERVAL_CLOSE_ONLY`). **Decision is low-risk; ship this one even if (1)/(2) are still being debated.** Open sub-question: do we also want a `CANCELLED_TERMINAL` zero-duration record so reports can answer "how many orders cancelled while in PV1 vs FILL vs FINAL?" without joining `order_cancellation.cancelledFromStatus` back to the workflow. **Recommended:** rely on `OrderCancellation.cancelledFromStatus` (it already carries the answer) — adding a terminal interval row is duplicate state. **(4) `ReopenForCorrection`** — rolls the order back to an earlier workflow state. Should close the currently-open interval and open a fresh `WAIT_BEFORE_*` for the `reopenToState` the command parameterizes. The next-kind decision is **data-driven by `reopenToState`**, which means this can't go in the static `COMMAND_STAGE_INTERVAL_TRANSITION` table — it needs either a per-state lookup (extend `applyCommandStageIntervalTransition` to read `reopenToState` from the command input) or the handler itself calls `transitionStageIntervals` directly with the resolved kind. **Recommended:** handler-direct call; keeps `applyCommandStageIntervalTransition` a pure command-name → static-pair lookup. **Acceptance criteria for the whole work item:** (i) `KNOWN_NON_SLA_COMMANDS` is empty; (ii) every workflow-mutating command has a code-reviewed SLA decision attached; (iii) the SLA report layer (when built) treats `HOLD_ACTIVE` and any `WAIT_AFTER_*_REJECT` as **excluded** from per-stage breach windows by default; (iv) dashboards differentiate "first-pass PV1 time" from "total PV1 time including reworks". **Blocker:** none — this is gated only on someone owning the dashboard semantics. Pick this up before building the breach evaluator (next bullet), because the evaluator needs to know which kinds count.
