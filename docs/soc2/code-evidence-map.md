## Why this document exists

The companion docs ([`controls-inventory.md`](./controls-inventory.md),
[`trust-service-criteria-mapping.md`](./trust-service-criteria-mapping.md),
[`evidence-inventory.md`](./evidence-inventory.md), and the higher-level
[`../security/control-matrix.md`](../security/control-matrix.md))
describe each SOC 2 Common Criterion and HIPAA safeguard at the level
of **policy + ADR + narrative + cadence**. That answers _what_ the
control is and _how_ Pharmax implements it conceptually.

This document answers a different question, the one a technical
auditor asks first:

> "Show me the code that enforces the control AND the test that
> proves it fires."

For each code-grounded control we ship, this map names — by exact
file path — the:

1. **Feature code** — the function, package, or migration that
   _implements_ the control.
2. **Schema / audit tables** — the Prisma model(s) or migration(s)
   that _store_ the evidence the control generates at runtime.
3. **Automated tests** — the `.test.ts` file(s) that _prove_ the
   control works (the test that fails if the control is silently
   removed).
4. **CI gate** — the `pnpm` script (run on every PR via
   `.github/workflows/ci.yml`) that pins the control's structural
   invariants.

A row whose **Automated tests** cell reads `— GAP —` is a control
that has no automated proof today and is explicitly tracked at the
bottom of this document. **There are 8 such gaps** as of v1.0,
each cross-referenced to a tracked follow-up slice.

This map is the engineering-grade companion to the higher-level
control matrix. The two documents stay in sync at the control-ID
level — `controls-inventory.md` is the source of truth for control
IDs and status; this document fills in the file paths.

## How to use this document

- **Auditor walkthrough.** Pick a control ID. Read the row. Open
  the named test file. Run `pnpm vitest run <test-path>` to confirm
  it passes. Open the named feature code to verify it implements
  what the control says it implements.
- **Engineer onboarding.** Find the table of the area you're
  working in (e.g. CC6 logical access). The rows are a map of "what
  invariants exist that I must not break."
- **Drift detection.** A future slice will add
  `scripts/soc2/check-evidence-map.ts` that parses this file,
  verifies every cited path exists, and runs each cited test
  individually. Until then, drift is caught by the
  general-purpose `pnpm verify` pipeline.

## Conventions

- Paths are relative to the repository root.
- `package/name/path.ts` references are the runtime path inside the
  package's `src/`, not the published path.
- A row that cites _multiple_ tests groups them with `+` —
  every cited test pins a different facet of the same control.
- Multiple rows per control are used when the same control has
  structurally distinct enforcement points (e.g. CC6.1 is
  identity / RBAC / RLS / MFA, four separate rows).
- The **Status** column mirrors `controls-inventory.md` —
  out-of-band status changes there propagate here on the next
  refresh.
