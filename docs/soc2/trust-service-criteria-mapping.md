# Trust Service Criteria → Pharmax Controls Mapping

This is the centerpiece of the SOC 2 evidence framework. For every Trust
Service Criterion Pharmax claims to operate against, this document names
the specific Pharmax control(s) that satisfy the criterion, the exact
implementation location (code path / ADR / migration / Terraform module
/ policy), the evidence source an auditor reads to confirm operating
effectiveness, the cadence at which evidence is captured, and the role
that owns the control.

Conventions:

- **Control ID** is `<criterion-id>-<n>` where `n` is the local
  enumerator for that criterion. A criterion may map to multiple
  controls; a control maps to exactly one criterion (the primary one
  the auditor reads it under). Cross-references are inline.
- **Implementation** points to the canonical code path, ADR, migration,
  Terraform module, or policy that constitutes the control. Auditors
  read the implementation column to answer "is the control designed?".
- **Evidence Source** points to the artifact an auditor reads to answer
  "is the control operating?". It is one of:
  - a database table (`audit_log`, `command_log`, `event_outbox`, etc.)
  - a script output (`scripts/soc2/<name>.ts` produces `evidence/...`)
  - a CI artifact (GitHub Actions run, SBOM, CodeQL report)
  - a vendor-provided artifact (SOC 2 report, BAA, signing receipt)
  - a signed PDF or attestation under `evidence/`
- **Frequency** is one of: `continuous` (control fires on every event),
  `daily`, `weekly`, `monthly`, `quarterly`, `annual`, `per-event`,
  `on-change`. Continuous controls are evidenced by query at the
  auditor's request; periodic controls are evidenced by the dated file
  produced on cadence.
- **Owner** is a role title (CTO, Engineering Lead, Security Officer,
  Compliance Officer, Workforce Lead, Finance Lead). Names are not
  recorded here; names live in the org chart and the policy headers.

Status: every row in this table reflects the engineering state as of
the most recent commit. The status vocabulary lives in
[`controls-inventory.md`](./controls-inventory.md) (`Implemented`,
`Partial`, `Planned`, `N/A`). This file is the **map**; the inventory
is the **catalog**.

## Maintenance

This file is refreshed:

- On every ADR addition or status change (cross-reference the new ADR
  in the relevant row).
- On every material change to the implementation (e.g. a new control,
  a deprecated control, a vendor swap).
- During the annual policy review (rollup) — confirm every row's
  implementation still points to live code and that every Evidence
  Source still produces.

A drift between this file and the codebase is a control deficiency; it
gets logged in the risk register and remediated in the next sprint.
