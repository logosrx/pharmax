# SOC 2 Auditor-Readiness Checklist

A numbered checklist a new CISO can walk through end-to-end before
engaging a SOC 2 auditor for a Type 2 examination. Each item links to
the canonical source of truth in the repo or the SOC 2 evidence vault.

## 1. Governance and policy

- [ ] **1.1** A board-approved Information Security Policy exists,
      versioned, with annual review date in the next 12 months.
- [ ] **1.2** An Acceptable Use Policy exists and every employee has
      signed it as part of onboarding.
- [ ] **1.3** A Risk Assessment was performed in the last 12 months,
      with documented mitigations and acceptance for residual risks.
- [ ] **1.4** A Vendor Management Policy exists; every critical
      sub-processor has a current BAA (HIPAA) or DPA (GDPR) on file.
- [ ] **1.5** An Access Control Policy exists. Cross-reference:
      `docs/policies/access-control-policy.md`.
- [ ] **1.6** An Incident Response Policy exists. Cross-reference:
      `docs/INCIDENT_RESPONSE.md`.

## 2. People and access

- [ ] **2.1** Every active employee has a background check on file.
- [ ] **2.2** Security training has been completed in the last 12
      months by every employee with production access. Records exist
      with completion dates and quiz scores.
- [ ] **2.3** Termination checklist exists and was followed for every
      departure in the audit period (account disable within 24 h,
      device collected, secrets rotated where applicable).
- [ ] **2.4** Quarterly access reviews are landing in the evidence
      vault under `evidence/<period>/CC6.2/`. The reports are
      generated by `scripts/security/run-access-review.ts` and
      signed by the assigning manager.
- [ ] **2.5** No production account is shared between humans. Every
      operator action traces to one `user.id`.
- [ ] **2.6** MFA is enforced for every operator role gated on
      `BillingManager`, `OrgAdmin`, and any role granting
      `patients.crypto_shred` or `audit.read`.

## 3. Engineering controls

- [ ] **3.1** All production changes flow through pull requests with
      at least one reviewer. No direct pushes to main.
- [ ] **3.2** CI runs `lint`, `typecheck`, `prisma:validate`,
      `check:migrations`, `check:schema`, `check:commands`, and the
      full `vitest` suite on every PR. PRs cannot merge with a red
      check.
- [ ] **3.3** Every critical command writes `command_log`,
      `order_event`, `audit_log`, and `event_outbox` inside one
      transaction. Verified by `scripts/check-command-files.ts`.
- [ ] **3.4** Every migration includes an RLS policy. Verified by
      `scripts/check-migration-rls.ts`.
- [ ] **3.5** Workflow-state mutation outside a command handler is
      forbidden by code rules and reviewed in code review.
- [ ] **3.6** PHI is envelope-encrypted at rest. KMS data keys are
      rotated per `docs/RUNBOOK.md` "Rotating a KMS data key".
- [ ] **3.7** Audit chain is verified at least monthly via
      `scripts/security/verify-audit-chain-all-orgs.ts`. Output
      committed to evidence vault.
- [ ] **3.8** Daily Merkle root signing is running in production —
      see ADR-0024 and the worker loop in
      `apps/worker/src/security/daily-merkle-root-loop.ts`.
- [ ] **3.9** Nightly security digest is being delivered to the
      security distribution list — see ADR-0024 and
      `apps/worker/src/security/nightly-security-digest-loop.ts`.

## 4. Logging, monitoring, and alerting

- [ ] **4.1** Structured logs from every process ship to a long-term
      log aggregator with at least one-year retention.
- [ ] **4.2** Sentry receives every `logger.error(...)` and uncaught
      exception. Allowlist scrubber active. See `docs/OBSERVABILITY.md`.
- [ ] **4.3** Alert routes from log aggregator and Sentry to PagerDuty
      (or equivalent) for SEV1/SEV2 patterns. On-call rotation
      published.
- [ ] **4.4** No PHI appears in logs, Sentry events, or alert
      messages. Spot-checked monthly.
- [ ] **4.5** Idle/heartbeat events are recorded for productivity
      analysis without recording keystrokes, screenshots, or
      personal-device activity. See `.cursor/rules/03-sla-performance.mdc`.

## 5. Infrastructure

- [ ] **5.1** Production infra is defined in Terraform. The Terraform
      state is stored in a versioned remote backend with locking.
- [ ] **5.2** Postgres backups are configured with point-in-time
      recovery for at least 14 days. Restore exercises performed in
      the last 12 months. See `docs/RUNBOOK.md` "Restoring from
      backup".
- [ ] **5.3** S3 audit-archive bucket exists, has Object Lock in
      COMPLIANCE mode, default retention matches retention policy,
      and only the worker's IAM role can write to it (per ADR-0024).
- [ ] **5.4** KMS asymmetric signing key for the Merkle job exists,
      `KeyUsage = SIGN_VERIFY`, has its own IAM policy granting
      `kms:Sign` to the worker role only.
- [ ] **5.5** Production VPCs deny ingress except via the WAF + ALB.
      Security groups reviewed in the last 90 days.

## 6. Vendor and BAA inventory

- [ ] **6.1** BAA tracker is current. Counterparties include at
      minimum: Postgres provider, Clerk, Stripe, Sentry, log
      aggregator, EasyPost (if any PHI flows through tracking
      events).
- [ ] **6.2** Every counterparty has been onboarded via the Vendor
      Management Policy: security questionnaire on file, BAA/DPA
      signed, custody owner identified.

## 7. Testing and validation

- [ ] **7.1** Annual external penetration test completed. Scope
      doc: `docs/compliance/pentest-scope.md`. Report committed to
      evidence vault. Findings remediated per documented SLAs.
- [ ] **7.2** Annual disaster-recovery drill performed. Postmortem
      documented. RTO/RPO targets met or revised with rationale.
- [ ] **7.3** Annual tabletop exercise on incident response
      performed. Cross-functional (engineering + ops + legal).
- [ ] **7.4** Annual review of permission registry
      (`packages/rbac/src/permissions.ts`). Each permission still
      needed; deprecations documented.

## 8. Compliance posture

- [ ] **8.1** Public privacy policy is current. Last review date in
      the page footer.
- [ ] **8.2** HIPAA Security Risk Assessment performed in the last
      12 months. Findings tracked to closure.
- [ ] **8.3** SOC 2 Type 1 report obtained (if not yet Type 2).
- [ ] **8.4** Evidence repository under
      `evidence/<period>/` is current and complete per
      `docs/compliance/evidence-collection-guide.md`.

## 9. People-facing readiness

- [ ] **9.1** A named Security Officer + Privacy Officer per
      `45 CFR 164.308(a)(2)`.
- [ ] **9.2** Org chart with engineering escalation paths, on-call
      rotation, executive-sponsor map.
- [ ] **9.3** Customer-facing security collateral up to date:
      Security FAQ, Trust Center page, third-party attestations.

## 10. Pre-engagement final pass

- [ ] **10.1** Re-run every script under `scripts/security/` for the
      audit period and confirm exit code 0.
- [ ] **10.2** Confirm the auditor's portal account is provisioned
      with the minimum required access (read-only) and that the
      access is time-bound.
- [ ] **10.3** Schedule the kickoff meeting at least two weeks
      before the period closes.
- [ ] **10.4** Confirm that every open Critical/High finding has an
      explicit risk acceptance or a documented remediation date
      within the auditor's window.

---

If you have ticked every box and have evidence in the repo, you are
ready to engage a SOC 2 Type 2 auditor. If any item is unticked,
treat its row as a P0 backlog item before the kickoff.
